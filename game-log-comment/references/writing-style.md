# 游戏日志写作层

基于 `data/game/summary.md` 的 100 篇合格样本总结。

## 样本给出的明确倾向

- 游戏日志常见开头是通关后的总判断、路线 / 周目切口，或者直接从某个系统体验说起。
- 相比动画和书籍，游戏日志更常处理“系统 / 节奏 / 路线 / 文本”四者之间的关系。
- 很多合格样本会保留记录感，但真正站得住的文章不会沦为流水攻略。
- 剧透提示出现频率更高，尤其在路线作、视觉小说、叙事型游戏里。

## 推荐结构

### 1. 开头
优先从下面几种方式中选一种：
- 先说通关后的总判断
- 从某条路线 / 某个系统 / 某个卡点切入
- 从“为什么我会继续玩下去 / 为什么我中途想放弃”切入

### 2. 主体展开
建议按下面顺序挑 2 到 4 个维度推进：
- 核心玩法 / 系统反馈
- 流程节奏 / 难度曲线 / 地图或关卡设计
- 文本、角色、路线、章节结构
- 个人操作体验与情绪变化

### 3. 结尾
回到：
- 这段游玩经历最后留下了什么
- 它推荐给谁，不推荐给谁
- 问题到底毁掉了作品，还是只是拉低上限

## 语气要求

- 可以更具体、更记录型，但不要变成攻略备忘录
- 评价系统时必须回到整体体验，不要只列机制清单
- 可以直说问题，但要交代问题如何影响游玩过程

## 常见问题

- 把攻略、配装、流程记录当成长评主体
- 玩法、文本、路线、氛围各说各话，无法汇总成总判断
- 只说好玩 / 不好玩，却说不清楚好玩在哪、不好玩在哪
- 一上来剧透到底，没有结构和提醒
