# Bangumi 通用写作层

让 Bangumi 长日志更像站内成稿，而不是公众号稿、说明书或流水账。

## 适用范围

- 动画 / 书籍 / 游戏日志
- 剧情梳理
- 角色 / 章节 / 路线分析

## 不负责

- 替用户捏造亲身体验
- 在素材不足时杜撰剧情、设定、路线或章节细节
- 把内容写成营销文或平台运营文案

## 素材优先级

1. 用户提供的笔记、摘录、观感、剧情点
2. 用户明确指定的范围（某集 / 某卷 / 某章 / 某路线）
3. 公开可得的剧情简介、设定、条目介绍
4. 仍不足时，只写结构化草稿，不假装完整分析

## 起稿顺序

1. 先判断是“评论向”还是“剧情向”
2. 评论向：先给判断，不先堆百科背景
3. 剧情向：先交代要整理 / 分析的是哪一段，再进入正文
4. 正文按“判断 / 问题 → 展开 → 细节 / 场景 → 回扣主旨”推进
5. 每段只推进一个主问题，不把复述、判断、经历全塞进同一段
6. 结尾回到“这部作品最后留下了什么”或“这段剧情说明了什么”

## 推荐骨架

### 评论向

- 标题
- 开头：快速抛出总判断或核心感受
- 第一部分：作品最抓人的点是什么
- 第二部分：为什么成立，展开到结构、角色、主题、系统或文本层
- 第三部分：个人体验、情绪余韵、适用边界或缺点
- 结尾：回收全文，让评价落地

### 剧情向

- 标题
- 开头：交代剧情范围与核心问题
- 第一部分：必要的剧情整理（只保留跟后文相关的）
- 第二部分：分析关键冲突 / 角色关系 / 主题推进
- 第三部分：这段剧情为何重要，或如何影响整部作品
- 结尾：回到这段剧情真正留下的意义

## 行文规则

- 先判断，再论证
- 剧情整理只保留分析需要的部分，不写流水账
- 多写“为什么这点有效 / 无效”
- 可以有个人口吻，但别碎成聊天记录
- 每段只讲一个意思
- 段落之间要有过渡，不要硬切
- 不为显得深刻而堆空话
- 不为像评论而故意端着

## 通用质检

- 标题是否有明确落点，而不是空泛抒情
- 开头是否尽快交代判断或剧情范围
- 是否至少有 3 个自然段的有效展开
- 是否有具体细节支撑判断
- 涉及剧情时，是否只使用了有根据的剧情点
- 结尾是否能收住，而不是突然中断
